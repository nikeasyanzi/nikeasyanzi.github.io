Sometimes it has to be perfect. Other times, you just need to kick off the game and let people play.

最近剛加入新東家的新團隊做OS Core Team Development Manager

這個團隊必須要跟公司的行銷團隊、世界各地的廠商做溝通。

這對我來說是一個充滿不確定性的挑戰，因為與我搭配的 Project Manager 也是新人。

雖然老闆有跟我說明自己的角色與定位，但我真正煩惱的是尺度的拿捏。

近期的任務是要在四週內產出一個年度 OS 導入計劃，讓其他部門的硬體團隊做參考。

當下遇到了幾個困難：

一來，因為我們的 proposal 必須仰賴其他團隊給予我們資料，但是其他團隊也遭遇相當規模的組織改變，所以我們在第一時間取得資料是有一些困難的。

二來，這個 proposal 也是有期限時效限制，在資料取得有困難的情況下，要在短時間內做出一份像樣的報告，並不是一件容易的事。

三來，這個 proposal 必須給 OS core team 其他成員審閱才得以進行。這些審閱的同事往往資歷深厚，身為新進人員的我們，很擔心自己寫出來的 proposal 並不到位。

這讓我思考，要多完美才能拿出來討論？太早拿出來怕顯得不專業，太晚拿出來又怕方向錯誤。

我後來下了一個大膽的決定：參考前幾次的報告作為範本，我將手邊已確定的資料提早兩週發布初版，讓核心團隊能提早審閱，而不是等到最後一週才讓大家檢視。

這樣一來，我們有更多時間可以收集其他資深成員的反饋，並適時調整與修正。
最後我們順利如期完成報告，取得核心團隊共識，讓計畫過關，成為全公司約 60 個硬體的設計指引。我的初登板也給資深成員留下了不錯的印象。

## 反思 (Reflection)

這件事情給我很大的刺激。雖然工作上我們都力求完美，但是在有限資源或者不可抗力的因素底下，我們能做的就是當下最好的決定，並且適時的收集他人的反饋讓成果更好。
